Building Your SINDy Model

Follow these steps to discover the underlying physics of your data.

01. Data Preparation

Upload your system measurements in .csv format. The data must be organized as a time-series state-space matrix.

Required Structure:

  • First Column: Strictly increasing time values \( (t) \).
  • Subsequent Columns: State variables \( (x_1, x_2, \dots, x_n) \).

⚠️ Ensure no missing values (NaN) and constant sampling intervals if possible.

Example of a properly formatted state-space CSV.

02. Parameters Suggestion

When data is uploaded, the module will read the data and come up with recommendations:

  • Library Type: Polynomial vs Fourier based on system behavior.
  • Degree: Estimated from linearity analysis of the derivatives.
  • Threshold: Estimated from the noise level in your data.

💡 These are starting points — you can always tune them manually.

AI Suggester analyzing time series, phase portrait, and FFT.

03. Train / Validation Split

Use the Split Slider to define how much data the algorithm will use during the discovery phase.

  • Training Set: Used to find the sparse coefficients \( \Xi \). Derivatives \( \dot{X} \) are computed on the full dataset first, then randomly split.
  • Validation Set: The remaining points, used to check the model on unseen data.

⚠️ SINDy loss is based on residuals of \( \dot{X} \), not \( x(t) \) directly. Train/Validation metrics reflect derivative-space accuracy.

04. Model Configuration & Training

Define the Mathematical Search Space. SINDy will find the sparsest combination of these functions to describe your data.

Library Type:

Choose Polynomial for most physical systems. Use Fourier only if you know the governing equations involve \( \sin/\cos \) terms.

Polynomial Degree:

Higher degrees (e.g., \( x^3 \)) allow more complex interactions but increase risk of noise fitting. Use the AI Suggester as a starting point.

Threshold \( (\lambda) \):

The sparsity controller. Higher \( \lambda \) prunes small coefficients → simpler equations. Too high and important terms get removed.

💡 Run multiple configurations and compare them in the History Leaderboard.

Effect of \( \lambda \) on equation sparsity.

05. Testing

A good SINDy model should generalize beyond the training trajectory. In the Test Tab, verify this by simulating the discovered equations from entirely new initial conditions.

  • Load Test CSV: A separate dataset generated from the same system but with different \( x_0 \). Do not use any portion of this for training to avoid overfitting.
  • Numerical Integration: The model uses solve_ivp (RK45) to forward-simulate from the test initial condition.
  • RMSE on \( x(t) \): Unlike training metrics (on \( \dot{X} \)), test RMSE is computed directly on the state trajectories.

⚠️ A model can have low training RMSE on \( \dot{X} \) but still diverge when forward-simulated. Always test!

True trajectory (dots) vs SINDy simulation (line) from a new \( x_0 \).

06. Predict Future Trajectories

Once you have a validated model, use the Predict Tab to simulate the system into the future from any initial condition of your choice.

Select Model:

Choose from any run in your History Leaderboard. Pick the one with the best validation and test metrics.

Set Initial Conditions \( x_0 \):

Enter comma-separated values for each state variable. Variable names are shown as hints based on your training data.

Prediction Horizon:

Set how far into the future (in seconds) to simulate. Longer horizons accumulate more integration error.

💡 The discovered equations are valid physics — they can predict behavior far beyond the original data range.
