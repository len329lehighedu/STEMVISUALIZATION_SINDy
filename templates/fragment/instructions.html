Building Your SINDy Model

A closer look at each tab of the module — what every control does and why.

Data Source

Bringing In Your Data

Start from the Select System dropdown. Pick one of the pre-set systems to test SINDy's ability or upload your own data file. The file must have time in the first column and one state variable per subsequent column — \( t, x_1, x_2, \dots, x_n \).

When the file is uploaded, the module also runs the System Suggestion in the background: it analyzes the linearity, periodicity, and noise level of your data and pre-fills the hyperparameter controls below with a reasonable starting point. These are only suggestions, as they can be wrong.

A correctly formatted state-space CSV — time in column 1, states after it.

Model Configuration

Choosing How SINDy Learns

Train/Validation set up & Split Strategy

Derivatives \( \dot{X} \) are always computed once on the full trajectory first — the split only decides how the resulting \( (X, \dot{X}) \) pairs are divided afterward. Three strategies are available, and they test different things:

Random Sampling

Every point is shuffled independently into train or validation.

Random Sampling example.

Time-based

Training uses the earlier portion of the trajectory, validation the later portion.

Time-based example.

Random Block

Splits by contiguous random blocks instead of single points.

Random Block example.

Library, degree, and sparsity threshold

The Library dropdown sets the candidate function space: Polynomial, Fourier, or Combined (both at once). Degree / Harmonics controls how high the polynomial power (or Fourier harmonic) goes — higher values allow more complex interactions but risk fitting noise.

The Sparsity Threshold \( (\lambda) \) prunes small coefficients: higher \( \lambda \) means a simpler, sparser equation, but push it too far and real terms get removed too. Because the slider steps in coarse increments of \( 0.005 \), there's also a plain text field next to it — type an exact value (e.g. \( 0.0347 \)) and the slider updates to match, and vice versa.

Training Results

Reading the Leaderboard

Every time you press Train, a new row is appended below, so you can compare configurations side by side. Clicking a row re-renders that run's plots without re-fitting anything; the Delete button removes a row.

Column Meaning
Data File / Split Type Which dataset and which split strategy produced this run.
Library / Degree / Threshold The exact configuration used, for reproducibility.
Train Metrics / Val Metrics \( R^2 \), RMSE, and MAE on the training vs. validation split — computed on \( \dot{X} \), not \( x(t) \).
RMSE Diff \( | \text{RMSE}_{\text{train}} - \text{RMSE}_{\text{val}} | \). A large gap is an early warning sign of overfitting.
Identified Equations The actual sparse ODE system SINDy recovered for this run.

Model Diagnosis

Residual Diagnostics

A residual is simply the leftover error the model didn't explain: for each state variable, \( \text{residual} = \dot{x}_{\text{true}} - \dot{x}_{\text{predicted}} \), computed across the full dataset (train + validation combined — so these numbers aren't directly comparable to the Train/Val metrics in the leaderboard above). If SINDy has found the right terms, whatever is left over should look like structureless noise. If it hasn't, the leftover error still carries a pattern — and that pattern tells you what's missing.

Residual vs. time

A well-fit model's residuals should scatter randomly around zero with no visible trend. A repeating wave or drift in the residual means the library is missing a term that captures that behavior

Residual vs time example.

Residual FFT (frequency content)

Same residual, viewed as a frequency spectrum. A flat spectrum with no dominant peak confirms the error really is noise. A sharp peak means there's periodic structure left over.

Residual FFT example.

True vs. predicted derivative

Each point plots \( \dot{x}_{\text{predicted}} \) against \( \dot{x}_{\text{true}} \) for one sample. A perfect model places every point on the \( y = x \) diagonal. Curvature or fanning away from that line signals bias or error.

True vs. predicted derivative example.

Reading the stats line

\( R^2(\dot{X}) \)
Fraction of derivative variance explained by the model, per variable — same idea as leaderboard \( R^2 \), but on the combined dataset.
SNR (dB)
Signal-to-noise ratio of the residual. Higher means the leftover error is small relative to the signal itself.
Autocorrelation
Whether consecutive residuals are correlated with each other. Close to zero supports "random noise"; a high value supports "still-structured error".
Model & Test Data

Choosing What To Test

Start from the Select Model dropdown — it lists every run you've trained on the Train tab, labeled Run #N. Picking one loads a snapshot of that run's discovered equations, not a live reference back to the Train tab.

For the bundled pre-set systems, the tab automatically fills in a matching pair of held-out test CSVs — trajectories generated from different initial conditions than the training file, so the check is genuinely out-of-sample. If the selected run was trained on a custom-uploaded dataset, the CSV dropdowns are swapped for two upload fields instead.

Tip. Test Run 2 is optional — leave the second dropdown on (none), or skip the second upload, to test with a single trajectory.

Model dropdown plus the two test-run file selectors.

Running The Test

Forward Simulation, Not Point Evaluation

The Train tab's \( R^2 \) is computed pointwise on \( \dot{X} \) — it never checks whether the equations stay stable when actually run forward. This tab does: pressing TEST takes the discovered ODE system and numerically integrates it with SciPy's solve_ivp (RK45), starting only from the test file's initial condition \( x(0) \) and stepping forward across its full time span. Any error in the discovered coefficients compounds over time exactly the way it would in a real deployment — which is why a model with a great training \( R^2 \) can still diverge badly here.

Each of up to two test trajectories gets its own overlay plot: the true, measured points as a scatter, and the SINDy-simulated trajectory as a solid line. A good model's line should track directly through the scatter for the full duration; a line that peels away partway through is a sign of instability or a missing term, not just noise.

Overlay plot — dots are measured data, the line is the forward-simulated trajectory.

Test Results

Reading The Metrics Table

Every completed test run appends its per-variable metrics below the plots — Run 1 and Run 2 (if used) both land in the same table, so they're easy to compare side by side.

Column Meaning
Test Run Which of the two test trajectories (Run 1 / Run 2) this row belongs to.
Model Which trained run (e.g. #3) produced the equations being tested.
Var The state variable this row's metrics apply to.
RMSE Root-mean-square error between the true and simulated trajectory, in \( x(t) \) space — comparable across runs on the same system.
R² Fraction of trajectory variance explained by the simulation. Unlike the Train tab, this is computed on the integrated path, so it directly reflects long-horizon stability.
When Something Goes Wrong

Reading a Failed Run

Every test run is validated before it's simulated, so failures come back as a plain-language message above the TEST button instead of a crash: a mismatched number of state variables between the test file and the trained model, missing/NaN or infinite values in the CSV, or solve_ivp failing to converge on a genuinely unstable system all produce their own specific error.

A successful run replaces that message with a green confirmation instead — if you don't see it, nothing was plotted yet.

Note. A variable-count mismatch usually means the test CSV's columns don't line up with the columns the model was trained on — check the state variable order first.
